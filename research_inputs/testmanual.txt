## 0. Purpose of this manual

This manual is the operational source of truth for QLMG backtesting work. It replaces the prior addendum-plus-old-manual format and folds the current project state into one document.

It is written for both the human operators and the backtesting AI agent. If a future prompt conflicts with this manual, the prompt must explicitly say so and explain why.

The manual has two jobs:

1. Prevent fake alpha from mechanical backtest errors.
2. Prevent the project from getting stuck forever waiting for data that will not be bought.

Every future run must move a hypothesis into one of these states:

```text
progress_with_current_data
redesign_to_less_depth_sensitive
needs_capture_substitute
candidate_library_only
discard_current_translation_no_vendor_path
blocked_by_protocol_issue
```

`waiting_for_vendor_data` is not an allowed terminal status.

## 1. Current project state

The project has completed a mechanical QA reset. Earlier QLMG research found useful hypotheses, but some prior outputs used unsafe evidence paths, including:

- placeholder or synthetic controls;
- duplicated/copy-like control summaries;
- projected internal-validation means treated as event R;
- summary rows used as if they were event-level trade ledgers;
- future-aware regime labels;
- full-sample thresholds;
- funding and mark proxies treated too strongly;
- promotion labels not backed by real event-level evidence.

Therefore, older labels such as:

```text
prelead
confirmed
stress_survives
beats_controls
targeted_execution_data_prelead
a2_a3_tier1_prelead_confirmed_train_only
```

are not rankable unless rebuilt under the current evidence contract.

Old results remain useful for:

- hypothesis discovery;
- infrastructure lessons;
- candidate-library preservation;
- identifying failure modes;
- selecting families to revisit.

Old results are not sufficient for:

- strategy promotion;
- validation;
- live readiness;
- final ranking;
- position sizing decisions.

## 2. Business objective and risk posture

The firm is small and expects small account sizes. The goal is aggressive account growth from genuine edge, not institutional capacity.

This creates several consequences:

- Capacity is secondary.
- Tier C small but tradable perpetuals remain strategically important.
- Execution quality, fill realism, liquidation risk, funding, data quality, and bad-wick avoidance are primary.
- Aggressive portfolio overlays are allowed in diagnostics, but never as a substitute for edge.
- Sparse or rare-regime variants can be preserved as portfolio-sleeve candidates.
- Live risk should increase only after evidence improves, not because a backtest looks exciting.

Small account does not mean reckless leverage. It means the account can trade smaller-capacity opportunities with smaller notional.

## 3. No-vendor-data policy

The project will not buy historical vendor data such as Tardis, Kaiko, Amberdata, CoinAPI, Coin Metrics, or similar paid historical depth/trade/liquidation feeds.

This is now a hard operating constraint.

If a strategy appears to require paid historical vendor data, it must be assigned one of these statuses:

```text
progress_with_current_data
needs_capture_substitute
redesign_to_less_depth_sensitive
discard_current_translation_no_vendor_path
preserve_hypothesis_generate_new_variant
candidate_library_only
```

A family cannot remain indefinitely blocked because vendor data is missing.

If the only plausible next step is paid vendor data, the current translation must be redesigned, replaced by a capture substitute, or discarded. The broader hypothesis can remain in the candidate library if the mechanism remains coherent.

## 6. Hard rejection versus soft cap

Do not reject hypotheses too quickly.

Hard rejection is allowed only for evidence or logic failure, such as:

- lookahead or point-in-time violation;
- projected or summary metrics used as event R;
- no event-level trade ledger where one is required;
- impossible fills or touch-fill dependence in Tier 1;
- future liquidation labels or future winners/losers used as filters;
- event before the instrument was tradable;
- current-only taxonomy used as historical sector truth;
- arithmetic mismatch between event rows and reported metrics;
- universally negative results across clean current translations and controls.

Soft cap or preserve labels should be used for:

- sparse event count;
- single-month concentration;
- single-regime concentration;
- sample-limited mechanisms;
- failed CPCV because independent folds are too few;
- mark or funding exactness cap;
- data-tier limitation;
- execution-depth limitation;
- current broad translation negative but mechanism still coherent.

Allowed preservation labels:

```text
portfolio_sleeve_candidate
rare_regime_sleeve_candidate
sample_limited_sleeve_candidate
feature_overlay_candidate
mechanism_preserved_current_translation_weak
current_translation_rejected_only
candidate_library_only
```

Use `family_rejected` only when multiple clean no-vendor translations have failed and the economic mechanism itself is no longer coherent.

## 17. Funding and mark treatment

Funding and mark must be explicit in every event-level ledger.

Rules:

- If no funding timestamp is crossed, set `funding_exact=true` and `funding_R=0`.
- If funding is crossed and exact funding exists, compute side-aware cashflow.
- If funding is crossed and exact rate is unavailable, set `funding_exact=false`, run adverse funding sensitivity, and cap the label.
- Repeated 5m funding fields are context/proxy unless proven to correspond to exact settlement events.
- Mark price is required for liquidation logic.
- If mark is missing and last-price proxy is used, set `mark_proxy_used=true` and cap the label.
- Proxy funding or proxy mark may support screening, but not final validation.

## 19. Risk and aggressive overlays

Use risk-based sizing.

```text
risk_amount = account_equity * risk_pct
stop_distance_pct = abs(entry - stop) / entry
position_notional = risk_amount / stop_distance_pct
initial_margin_required = position_notional / leverage
gross_leverage = gross_position_notional / account_equity
```

Research overlays may test aggressive small-account risk, including 5%, 10%, 15%, and 20% risk per trade, but these overlays are diagnostic only until a candidate has clean evidence.

Aggressive overlay results cannot promote a weak or unproven strategy.

Any overlay must report:

- ruin frequency;
- >50%, >75%, >90% drawdown events;
- max drawdown;
- liquidation count;
- skipped trades;
- capital path under 200 / 500 / 1000 USDT starts.

## 21. Parameter search

Use coarse randomized or Sobol search first, not full grid explosion.

Process:

1. Write a strategy contract.
2. Define admissible parameter ranges.
3. Register every candidate before scoring.
4. Generate event-level trade ledgers.
5. Apply real controls.
6. Apply Tier-appropriate stress.
7. Score by robustness and evidence quality, not raw Sharpe.
8. Preserve sparse sleeves separately from standalone candidates.
9. Freeze candidates before any later holdout work.

Do not manually tweak thresholds after seeing validation results.

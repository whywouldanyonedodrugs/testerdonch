# Point In Time Sector Seeds for Crypto Perpetual Theme Ignition

## Executive summary

A usable point-in-time sector map can be built from public sources, but only if the dataset is split into three classes of evidence. The strongest class is **dated artifacts**: research reports, dated methodology documents, whitepapers, and protocol launch posts that were public at the time. The second class is **stable functional documentation**: official protocol docs that make a protocol’s core function clear, but were published later or are undated current pages; these are often good enough for a stable primary sector but usually only at **medium** confidence. The weakest class is **current-only tagging systems** such as live category pages on CoinGecko, CoinMarketCap, and DefiLlama; they are useful for discovery and QA, but unsafe as sole historical truth because the live endpoints do not expose historical classification snapshots and because category assignment rules can change over time. citeturn29view0turn29view2turn30view0turn31view0turn33view0turn32view0

For a sector/theme ignition strategy on crypto perpetuals, the highest-confidence reusable data are the **stable functional sectors**: currencies/payments, smart-contract platforms, rollups/L2s, oracle/data, DeFi lending/DEX, liquid staking, tokenized/RWA protocols, and AI-native protocols when AI is part of the protocol’s core design rather than a later market label. Dated research frameworks from Binance Research, CoinGecko, FTSE/Grayscale, and Bitwise are especially useful because they show what categories were publicly legible at specific times. citeturn21view0turn21view4turn21view5turn24view0turn27view0turn27view2turn28view0

The parts that are most dangerous for historical backtesting are **ecosystem tags** and **narrative tags**. CoinGecko’s own methodology states that ecosystem tags such as “Ethereum Ecosystem” or “Solana Ecosystem” are auto-assigned from contract addresses, while category tags such as “AI,” “DeFi,” and “RWA” are manually requested market labels and subject to periodic review. That makes current CoinGecko tags a good enrichment layer, but not safe PIT truth unless you archived them yourself or can recover them from dated reports or the Wayback Machine. CoinMarketCap exposes category endpoints, but its public docs show no historical date parameter for category membership, unlike its listings-historical endpoint. DefiLlama’s category pages are excellent for current DeFi protocol structure, but they are still live views unless separately archived. citeturn29view0turn29view2turn30view0turn31view0turn31view1turn33view0turn9search1

For B1 testing of “theme ignition leader continuation,” the most usable themes are: **DeFi summer 2020; alt-L1 / smart-contract rotation 2021; L2/Base and broader rollup rotation 2023–2024; AI tokens 2023–2025; Solana ecosystem and Solana memecoin waves 2023–2025; restaking 2024; TON/Telegram ecosystem 2024–2025; and RWA/tokenization 2023–2025.** These themes have reasonably clear public recognition points and liquid leader assets that were or became perpetual candidates. NFT/metaverse 2021 is usable, but more attention-driven and more vulnerable to narrow-asset distortions. Privacy rotations are real, but the evidence for broad, liquid, multi-asset breadth over 2020–2025 is weaker and exchange delistings create extra selection bias. citeturn25search12turn26search11turn34search8turn34search1turn34search5turn23view0turn24view0turn24view1turn21view3turn7news29turn35search15

## Practical schema and seed sector map

The schema below is appropriate for a backtest table, but the date fields should allow **coarse strings** like `<=2020-01-01`, `2023-03`, or `2023-03-23` so the dataset does not invent day-level precision that the sources do not actually support. That is safer than silently converting vague evidence into exact timestamps. The right rule is: **primary_sector should be functional and relatively stable; secondary_tags and ecosystem_tags may change over time and must carry lower confidence unless directly evidenced by a dated source.** citeturn29view0turn32view0turn27view2

## Historical theme ignition episodes

The best use case for this strategy family is not “buy all assets in a category.” It is: detect a **publicly legible breadth impulse**, measure which liquid names within that theme are actually confirming, and then trade the strongest leader only after the theme is visible in public information and market breadth. Dated report pages from CoinGecko and Binance Research are very useful here because they mark when a narrative had become recognizable enough to be discussed as a category rather than as isolated idiosyncratic moves. citeturn23view0turn24view0turn21view3turn35search15

## Recommended backtesting use

A good backtest should treat this dataset as **two different products**. The first is a **rankable PIT sector map** built from `primary_sector`, `sub_sector`, and only those `secondary_tags` whose start date is supported by a dated source. The second is an **annotation layer** for narratives and ecosystem tags. Mixing them is where most lookahead mistakes happen. citeturn29view0turn31view1turn33view0

The rankable PIT features should include:

| use class | fields | rule |
|---|---|---|
| PIT-rankable | `primary_sector`, `sub_sector`, `effective_start_utc`, `effective_end_utc`, `sector_confidence` | Use only rows with `high` or `medium` confidence and only after `effective_start_utc`. |
| PIT-rankable when dated | `secondary_tags` | Only if the tag has dated public evidence. Example: `RWA` on ONDO from 2023-01-10 is usable; `AI` on RNDR from a current aggregator page is not. |
| annotation only | `ecosystem_tags`, `current_only` tags, narrative popularity notes | Use for analysis and filtering, not as triggers or ranking inputs. |
| exclude | live aggregator categories without archive history | Exclude from historical feature generation. |

The most robust first-pass implementation is:

1. Use the sector map to define a **candidate universe**.
2. Use rolling return correlation / residual clusters to verify that a theme is actually moving as a group.
3. Require a **dated public theme event** or a **breadth threshold** before enabling the strategy for that theme.
4. Rank only assets that are both **eligible perps** and **in-theme by PIT sector map**.
5. Trade the leader, not the basket.

The parts that should be excluded from B1 are the current-only ecosystem and narrative tags from live pages, any backfilled category membership recovered only from 2026-level snapshots, and assets whose theme identity changed materially but lacks a dated first-public evidence point. That includes many “AI,” “RWA,” “ecosystem,” and “memecoin” labels on aggregators. citeturn29view0turn30view0turn33view0

### Open questions and limitations

This seed leaves some important names for phase two: DOT, NEAR, ATOM, FIL, GRT, IMX, DOGE, SHIB, XLM, BCH, ETC, ZEC, XMR, JASMY, HYPE, and TRUMP can be added with the same rules, but I have not expanded the machine-readable seed to the full watchlist here. The source base already supports doing that, especially for stable-function names and for memecoin/payment/privacy annotations. citeturn41search1turn41search2turn17search2turn17search3turn39search3turn40search2

There is also a structural limitation in public data: the strongest “theme popularity” evidence often comes from **reports** rather than **APIs**. That means a production historical build should archive the raw pages or reports you rely on, and ideally store the excerpt, timestamp, and parser version alongside each sector-row update. Otherwise reproducibility degrades quickly as sites revise page structures and category rules. citeturn9search1turn29view0turn30view0

## Backtesting-agent instructions

Use the following prompt with a backtesting agent:

> Ingest the PIT sector/theme seed table as a time-bounded classification dataset. Treat `primary_sector` and `sub_sector` as rankable only when `sector_confidence` is `high` or `medium` and the simulation timestamp is within `[effective_start_utc, effective_end_utc)`. Treat `secondary_tags` as rankable only when they are not `current_only`. Treat `ecosystem_tags` and all `current_only` fields as annotation only.  
>  
> For each day, build rolling co-movement clusters on the perp universe using returns, residual returns vs BTC and ETH, volume shock, OI shock, and funding shock. A theme is “ignited” only if both conditions are met:  
> `a)` at least one PIT-valid sector/theme bucket shows breadth across multiple assets, and  
> `b)` the rolling cluster confirms abnormal within-theme co-movement.  
>  
> Once a theme is ignited, rank eligible assets by a leader score combining relative strength, liquidity, tradability, breadth-confirmation, and distance above theme median. Go long only the strongest liquid leader after confirmation. Never backfill current-only labels into history. Never infer theme validity from present-day category pages unless the dataset contains a dated start record.
